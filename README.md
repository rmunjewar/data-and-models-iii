# TeamPulse

AI-powered employee retention intelligence. Upload an HR/sentiment CSV, get
turnover-risk predictions per employee, and a personalized 30-day action plan
generated by an LLM — tailored to your company's context.

```
        ┌──────────────┐    CSV     ┌────────────────┐   row    ┌──────────────┐
manager │  React UI    │ ─────────▶ │  FastAPI       │ ───────▶ │  Groq LLM    │
        │  (Vite, TS)  │ ◀───────── │  (model + API) │ ◀─────── │  llama-3.1   │
        └──────────────┘  JSON      └────────────────┘   JSON   └──────────────┘
```

There are three independent pieces in this repo:

1. **The Ridge regression model** (`model.ipynb`) — trains the turnover-probability scorer from a labelled HR dataset.
2. **The Groq LLM service** (`llm.py`) — a FastAPI backend that scores uploaded CSVs and generates per-employee retention recommendations via Groq.
3. **The React frontend** (`src/`, `index.html`) — a Vite + TypeScript SPA that ties everything together for the manager.

---

## 1. The Ridge regression model (`model.ipynb`)

### What it does
A Jupyter notebook that trains a scikit-learn **Ridge regression** model to
predict `turnover_probability_generated` (0–1) from ~17 numerical HR/sentiment
features per employee.

### Inputs
The training set is the public dataset
[BrotherTony/employee-burnout-turnover-prediction-800k](https://huggingface.co/datasets/BrotherTony/employee-burnout-turnover-prediction-800k),
trimmed and cleaned into a CSV with columns:

| Category | Columns |
|---|---|
| **Identity** | `employee_id`, `role`, `department`, `job_level` |
| **HR facts** | `tenure_months`, `salary`, `overtime_hours` |
| **Performance** | `performance_score`, `goal_achievement_rate`, `project_completion_rate`, `training_participation`, `role_complexity_score`, `career_progression_score` |
| **Wellbeing** | `satisfaction_score`, `stress_level`, `workload_score` |
| **Sentiment / collaboration** | `team_sentiment`, `email_sentiment`, `slack_activity`, `meeting_participation`, `collaboration_score` |
| **Free-text context** | `recent_feedback`, `communication_patterns`, `risk_factors_summary`, `turnover_reason` |
| **Labels / outputs** | `burnout_risk`, `left_company`, `turnover_probability_generated`, `persona_name` |

### Output
The notebook writes the enriched dataframe — including the predicted
`turnover_probability_generated` column — back to the `trimmed_data` CSV that
the FastAPI backend then consumes.

### How to (re)train
```bash
jupyter notebook model.ipynb
# run all cells
```
The trained model parameters are baked into the CSV (predictions are
pre-computed). For ad-hoc CSVs that *don't* contain
`turnover_probability_generated`, the FastAPI service falls back to a
lightweight normalized heuristic over the same features so the upload flow
still works without retraining (`predict()` in `llm.py`).

## 2. The Groq LLM service (`llm.py`)

### What it does
A FastAPI app that:

1. Accepts uploaded HR CSVs and runs the model scorer (`predict()`).
2. Builds a structured employee context dict.
3. Calls **Groq** (`llama-3.1-8b-instant`) with a strict JSON-mode prompt to
   classify risk and generate **3 actionable, 30-day recommendations**.
4. Tailors recommendations to the manager's company context (industry, size,
   manager notes) injected into the system prompt.

### Endpoints
| Method | Path | Body | Returns |
|---|---|---|---|
| `GET` | `/api/health` | — | `{status, model}` |
| `POST` | `/api/upload` | raw CSV text | `{count, average_turnover_probability, employees[]}` |
| `POST` | `/api/recommend` | `{employee, company_context}` | `RecommendationResponse` |

### `RecommendationResponse` schema
```json
{
  "employee_id": "SYN_00420291",
  "risk_level": "High",            // "High" >= 0.6, "Medium" 0.3–0.6, "Low" < 0.3
  "turnover_probability": 0.82,
  "key_risk_factors": ["Severe burnout risk", "Stress level 0.98", "Feels overwhelmed"],
  "recommendations": [
    "Schedule a 1-on-1 this week to reprioritize the workload together.",
    "Offer two flexible/remote days for the next month to recover.",
    "Enroll in stress management training and protect 2hrs/week for it."
  ]
}
```

### Configuration
Create a `.env` file next to `llm.py`:
```
GROQ_API_KEY=gsk_your_key_here
```
`.env` is gitignored. Get a key at <https://console.groq.com/keys>.

### Running locally
```bash
pip install -r requirements.txt
uvicorn llm:app --reload --port 8000
```
Sanity check: <http://localhost:8000/api/health> should return
`{"status":"ok","model":"llama-3.1-8b-instant"}`.

### Key implementation notes
- `predict()` passes through `turnover_probability_generated` if the CSV
  already contains it, otherwise computes a normalized weighted blend of
  positive risk factors (stress, workload, overtime) minus negative risk
  factors (satisfaction, sentiment, performance, etc.).
- `df.to_json()` is used for serialization to safely convert NaN → null.
- CORS is wide-open for local dev; lock it down before production.
- Free-text columns like `recent_feedback` and `risk_factors_summary` are
  passed verbatim to the LLM as part of the per-employee context, which
  significantly improves recommendation quality.

## 3. The React frontend (`src/`)

### Stack
- **Vite** + **React 18** + **TypeScript**
- **react-router-dom** for client-side routing
- **zustand** for global state (company context + uploaded employees)
- **recharts** for pie / bar / radar charts
- Plain CSS with a custom dark theme (`src/styles.css`) — no Tailwind, no UI library

### File layout
```
src/
├── main.tsx               # Vite entry, BrowserRouter
├── App.tsx                # Routes + route guards + Footer
├── store.ts               # Zustand store + risk bucketing helper
├── api.ts                 # fetch wrappers for /api/upload + /api/recommend
├── styles.css             # Dark theme, layout, utility classes
├── components/
│   └── Footer.tsx         # Sticky footer
└── pages/
    ├── Login.tsx          # Manager intake: company name, industry, size, notes
    ├── Upload.tsx         # CSV drag-and-drop → POST /api/upload
    ├── Company.tsx        # Dashboard: KPIs, pie, dept bar chart, tips, list
    └── Employee.tsx       # Drill-down: radar, peer bars, tips, LLM action items
```

### User flow
1. **Login** — manager enters company name, industry, size, and free-text
   notes that will be passed to the LLM as `company_context`. (No real auth.)
2. **Upload** — drag-and-drop a CSV. Body is POSTed to `/api/upload`. The
   response is stored in Zustand.
3. **Company dashboard** — `/company`
   - KPI strip: avg turnover risk, high-risk count, avg satisfaction, avg stress.
   - Risk distribution pie chart.
   - Average risk by department bar chart (color-coded green→amber→red).
   - Auto-generated insights & tips based on the team's metrics.
   - Sortable, paginated employee list (15 per page; sort by risk or name).
4. **Employee detail** — `/employee/:id`
   - Identity card: role, level, tenure (formatted as years/months), salary, big color-coded turnover %.
   - "Vs. team average" horizontal bars for stress, satisfaction, workload, overtime.
   - **Wellbeing snapshot** radar chart (satisfaction, sentiment, performance, collaboration, goals, calm, sustainable load).
   - Auto-generated quick tips from the employee's own metrics.
   - **LLM action items** — fetched on mount via `POST /api/recommend`,
     rendered as a chip list of risk factors plus three numbered cards.

### State, routing & guards
Zustand `useStore` holds `{companyContext, employees, averageTurnover}`.
`App.tsx` wraps protected routes in `<RequireContext>` and `<RequireData>`
guards that redirect back to `/` or `/upload` if state is missing — so a hard
refresh on `/employee/X` lands you at the login page rather than crashing.

### Running locally
```bash
npm install
npm run dev          # Vite on http://localhost:3000
```
Vite proxies `/api/*` to `http://127.0.0.1:8000` (the FastAPI backend), so you
need both processes running.

## End-to-end: run the whole thing

```bash
# 1. backend
pip install -r requirements.txt
echo "GROQ_API_KEY=your_key" > .env
uvicorn llm:app --reload --port 8000

# 2. frontend (separate terminal)
npm install
npm run dev

# 3. open
open http://localhost:3000
```

Drop a CSV from `~/Downloads/trimmed_data.csv` (or the repo's `trimmed_data`
file) into the upload page and click through.

---

## Deployment notes

- **Frontend**: builds to a static SPA via `npm run build`, deployable to any
  static host. Vercel works great — set `VITE_API_BASE` if you split the
  backend onto a different origin.
- **Backend**: the FastAPI app can be hosted anywhere that runs Python
  (Render, Fly, Railway, Vercel Python functions, a VM). Set `GROQ_API_KEY`
  as a secret env var. Lock CORS down to your frontend origin.
- **Data**: uploads are kept in browser memory only — there is no persistence
  layer. Add MongoDB or any DB if you need history across sessions.
